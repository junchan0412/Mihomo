# 桌面客户端差距分析

更新日期：2026-07-24。

本文把外部产品的公开资料与当前 Mihomo 的实现分开记录。竞品功能是资料事实；优先级与路线是本项目的判断，不代表竞品优劣排名。

## 已完成的基线

- 策略页与菜单栏可显示节点延迟、按延迟排序和筛选，并可按组或全量测速。
- 节点提供商可批量导入、分组、打标签，并与 Profile 的 `proxy-providers` 双向同步。
- Profile 远程刷新保留上次内容中缺失的 Provider，避免上游响应遗漏破坏本地关联。
- 侧栏支持可折叠分区、快捷收藏和统一背景。

## 已知缺口与优先级

### P0：数据安全与可解释性

1. Provider 同步保留 YAML 的注释、键顺序和原始格式。当前结构化合并会保留未知字段，但序列化后的格式可能变化。
2. 为同名 Provider 引入稳定的“来源 + 名称”身份，并在 URL、类型或路径不一致时显示冲突预览。当前按名称合并，跨 Profile 同名但不同来源时存在歧义。
3. 在写回 Profile 前提供变更摘要和单步撤销，特别是批量导入、关联切换与远程刷新后的合并。

### P1：高频操作效率

1. 菜单栏增加最近切换节点、收藏策略组和可选快捷键；搜索结果按当前选择、可用性和延迟排序。
2. 为策略组提供延迟历史、失败原因和最近测速时间，而不仅是单次数字。
3. 增加命令面板，统一页面跳转、配置切换、测速、刷新资源和开关网络能力。
4. 资源页支持基于分组和标签的多选批量启停、刷新、迁移和导出。

### P2：规则与系统体验

1. 增加按进程、域名、规则集和策略组串联的路由解释视图，说明某次连接为何命中某条规则。
2. 增加 Profile/覆写的结构化 diff、版本快照和按字段回滚。
3. 提供可导出的隐私化运行健康报告，帮助排查 TUN、DNS、系统代理和资源更新问题。

## 自检结论

本次完成了导航连续性、延迟决策和 Provider 生命周期的主路径，但 Provider 同步仍是最需要谨慎演进的区域：数据正确性、格式保真、冲突解释和可逆操作应优先于继续扩展更多字段或按钮。下一轮应先完成 P0，再处理更复杂的自动化和个性化功能。
