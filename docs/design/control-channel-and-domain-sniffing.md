# 核心控制与域名嗅探设计

## 为什么不再直接展示 Controller

Controller 是 mihomo 提供的 HTTP/WebSocket 控制接口，客户端依靠它读取流量、连接、日志、策略和运行状态。它是客户端与核心之间的内部控制通道，不是用户必须先理解和配置的代理模式。

参考 Sparkle 的处理方式，Mihomo 默认自动管理这条通道：

- 客户端始终通过 `127.0.0.1` 连接自己启动的核心。
- 运行配置中的 `external-controller` 和 `secret` 由应用统一写入，Profile 不再覆盖它们。
- 普通界面使用“刷新核心状态”“核心控制通道”等结果导向文案。
- 只有用户需要从其他设备管理此 Mac 时，才在设置的“远程访问”中显示监听地址、管理端口和访问密钥。
- 开启远程管理且没有密钥时，应用自动生成随机访问密钥。

远程管理和局域网代理相互独立：前者允许读取状态和切换策略，后者允许其他设备使用 Mixed/SOCKS 代理端口。

## 域名嗅探是什么

域名嗅探会检查连接的初始握手信息，从 HTTP Host、TLS SNI 或 QUIC 中识别域名。它用于解决应用直接连接 IP 时，域名规则无法匹配的问题。

它不会：

- 解密 HTTPS 正文；
- 读取账号或密码；
- 代替 DNS 查询；
- 扫描局域网设备。

## 信息架构

域名嗅探属于网络流量识别，因此放在“网络”工作区，而不是通用或高级设置：

- 网络概览提供状态、快速开关和“详细设置”入口。
- 独立“域名嗅探”分段解释作用并配置 HTTP、TLS、QUIC 端口。
- 识别方式使用用户语义描述 `parse-pure-ip`、`force-dns-mapping` 和 `override-destination`。
- 例外规则支持跳过/强制域名、来源地址和目标地址。
- 核心运行时，应用设置会明确执行“应用并重启核心”。

## 配置同步

当前 Profile 中声明的 `sniffer` 会在启用或刷新配置时载入网络工作区，因此 Profile 初始值优先于 App 默认值。

用户之后在网络工作区修改域名嗅探并应用时，变化会同步写回当前 Profile。JS Transform 和 YAML 覆写仍只影响最终 Runtime Config，不反向修改 Profile。Controller 的 `external-controller` 与 `secret` 仍是唯一由 App 强制管理的配置例外。
